3. DATABASE SCHEMA (13 TABLES)
================================================================================

TABLE: users

  - full_name (VARCHAR 100, NOT NULL)
  - email (VARCHAR 100, UNIQUE, NOT NULL)
  - phone (VARCHAR 20)

  - status (ENUM: active, inactive, pending, deleted - DEFAULT pending)

  - last_active (DATETIME)
  - deleted_at (DATETIME)
  - deleted_by (INT, FK to users)

  - created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP)

TABLE: stokvels

5. MIDDLEWARE
================================================================================

AUTHENTICATE (auth.js)
  - Extracts JWT from Authorization: Bearer <token> header
  - Verifies token using JWT_SECRET
  - Attaches { id, email, role } to req.user
  - Returns 401 if token is missing or invalid

REQUIRE ADMIN (auth.js)
  - Checks if req.user.role === 'admin'
  - Returns 403 Forbidden if not an admin

OPTIONAL AUTH (auth.js)
  - Same as authenticate but does not fail
  - Silently attaches user if token is valid
  - Continues without error if no token

UPDATE LAST ACTIVE (auth.js)
  - Updates users.last_active = NOW() for authenticated user
  - Non-blocking (errors are swallowed)

VALIDATE (validate.js)
  - Checks express-validator validation results
  - Returns 400 with field-level error details if validation fails

6. API ENDPOINTS
================================================================================

Base URL: /api
All authenticated routes require: Authorization: Bearer <jwt_token>

--------------------------------------------------------------------------------
6.1 AUTH ROUTES - /api/auth
--------------------------------------------------------------------------------

POST /api/auth/register
  - Registers a new user with status = 'pending'
  - Accepts: fullName, email, phone, password, selectedStokvel (optional)
  - If selectedStokvel provided, creates a join_request with status 'pending'
  - Notifies all admin users about new registration
  - Returns: { message, userId }

POST /api/auth/login
  - Authenticates user with email + password
  - Rejects deleted, pending, and inactive accounts with specific messages
  - Returns JWT token (24h default, 30d if rememberMe=true)
  - Returns: { token, user: { id, name, email, role, status } }

POST /api/auth/forgot-password
  - Generates 6-digit reset code with 15-minute expiry
  - Stores code in password_reset_tokens table
  - Sends code via email using sendPasswordResetEmail()
  - Never reveals whether email exists (security)
  - Returns: { message }

POST /api/auth/verify-code
  - Validates the 6-digit reset code against the database
  - Does not consume the code (can be verified multiple times)
  - Returns: { valid: true/false }

POST /api/auth/reset-password
  - Consumes the reset code and updates the user's password
  - Marks the token as used
  - Returns: { message }

GET /api/auth/me
  - Returns current authenticated user profile
  - Includes all active stokvel memberships with progress percentages
  - Returns: { id, name, email, role, profiles: [...] }

6.3 STOKVEL ROUTES - /api/stokvels
--------------------------------------------------------------------------------

  - Returns: [{ id, name, type, targetAmount, currentMembers, cycle, ... }]

POST /api/stokvels/:id/join-request (authenticated)

  - Admin users cannot join stokvels (403)

  - Checks for existing pending request (409 if already pending)
  - If previously rejected, updates existing row back to 'pending'
  - Notifies stokvel admins via in-app notification
  - Returns: { message }

--------------------------------------------------------------------------------
6.5 LOAN ROUTES - /api/loans (all authenticated)
--------------------------------------------------------------------------------

GET /api/loans
  - List own loans with pagination and filtering
  - Includes daysRemaining and maxLoanable per loan
  - Filters: stokvelId, status, profileId
  - Returns: { data: [...], pagination: { ... } }

GET /api/loans/download
  - Download loan history as PDF, Excel, or CSV
  - Query params: profileId, format (pdf/excel/csv)
  - Returns: binary file

GET /api/loans/stats
  - Loan statistics for current user
  - Active loans count and total principal amount
  - Repaid loans count and amount
  - Total amount ever borrowed
  - Returns: { activeCount, activeAmount, repaidCount, repaidAmount, totalBorrowed }

POST /api/loans/request
  - Request a new loan
  - BUSINESS LOGIC:
    * Loan limit = 50% of total contributions
    * Total contributions = current saved_amount + active loan principal
    * Only one active loan per stokvel allowed
    * Minimum loan amount: R1 (step: R0.01)
    * Auto-approved (status = active immediately)
    * 30-day repayment term
    * 30% flat interest rate
    * Principal deducted from saved_amount on borrow
  - Requires a saved card
  - Creates notification for user
  - Returns: { message, loanId }

POST /api/loans/:id/repay
  - Repay an active or overdue loan
  - BUSINESS LOGIC:
    * Principal returned to saved_amount (capped at target_amount)
    * Interest recorded as a loan_repayment contribution to the stokvel pool
    * Loan status set to 'repaid', repaid_date = now
    * User and stokvel admins notified
  - Returns: { message }

6.6 PAYMENT ROUTES - /api/payments
--------------------------------------------------------------------------------

POST /api/payments/webhook (UNAUTHENTICATED)
  - Paystack webhook handler
  - Verifies HMAC-SHA512 signature using PAYSTACK_SECRET_KEY
  - Handles 'charge.success' events
  - Confirms matching pending contributions
  - Updates saved_amount (capped at target)
  - Returns: 200 OK

POST /api/payments/initialize (authenticated)
  - Initialize a Paystack payment
  - Creates a pending contribution record
  - Calls Paystack API to get authorization URL
  - Amount sent to Paystack in kobo (cents, amount x 100)
  - Returns: { authorizationUrl, reference }

GET /api/payments/verify/:reference (authenticated)
  - Verify a Paystack payment by reference number
  - Confirms or fails the contribution based on Paystack response
  - Updates saved_amount on success (capped at target)
  - Idempotent (safe to call multiple times)
  - Returns: { status, amount, reference }

9. BUSINESS LOGIC SUMMARY
================================================================================

REGISTRATION FLOW:
  1. User registers -> status = 'pending'
  2. Optional: selects a stokvel -> join_request created
  3. All admins notified
  4. Admin reviews and approves -> user status = 'active', profiles created
  5. Approval email sent to user

JOIN STOKVEL FLOW:
  1. Existing user browses stokvels from dashboard
  2. Clicks "Request to Join" -> join_request created
  3. Stokvel admins notified
  4. Admin approves -> profile created, email sent
  5. Or admin rejects -> user notified

CONTRIBUTION FLOW:
  1. Member selects stokvel profile and amount
  2. Amount validated: >= R100, <= remaining target
  3. Payment via card (local) or Paystack
  4. Contribution created with status = 'pending'
  5. Admin confirms -> saved_amount updated (capped at target)
  6. Or Paystack webhook/verify confirms automatically

LOAN FLOW:
  1. Member requests loan from a stokvel
  2. Validation: max 50% of total contributions, one active loan per stokvel
  3. Total contributions = saved_amount + any active loan principal
  4. Loan auto-approved, principal deducted from saved_amount
  5. 30-day term, 30% interest
  6. On repayment: principal returns to saved_amount, interest goes to pool
  7. Interest tracked in "Interest Pot" on stokvel dashboard

PAYMENT PROCESSING (Paystack):
  1. Frontend calls /payments/initialize
  2. Backend creates pending contribution, calls Paystack API
  3. User redirected to Paystack payment page
  4. On success: webhook or /verify confirms the contribution
  5. saved_amount updated, notifications sent

ACCOUNT DELETION:
  - Admin can soft-delete (reversible) or permanently delete
  - Self-delete requires password, blocked for admins/users with active loans
  - Soft-delete: status='deleted', data anonymized, profiles deactivated
  - Restore: reactivates user and profiles
  - Permanent delete: cascading removal from database
